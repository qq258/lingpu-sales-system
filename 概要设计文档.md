# 手机销售后台管理系统 — 概要设计文档

## 1. 项目概述

### 1.1 项目目标
构建一套统一的轻量化手机销售后台管理系统，部署在局域网内，支持多个分店同时使用。各门店数据隔离（仅操作本店数据），支持门店间调货，老板可查看全部门店经营数据。

### 1.2 设计原则
- **统一部署**：一套系统覆盖所有分店，无需各自搭建
- **门店隔离**：通过 store_id 实现数据隔离，店员只能看到本店数据
- **实时共享**：SKU等基础数据全门店共享，库存数据按门店独立
- **简单直接**：销售就是"在xx时卖出xx型号手机xx台，收款xx元"
- **轻量化**：SQLite 数据库，单文件易部署易备份

## 4. 数据库设计

### 4.1 ER 关系

```
Store（门店） ─── 1:N ──── User（用户，用户归属门店）
Store ─── 1:N ──── Inventory（库存，按门店独立）
Store ─── 1:N ──── SaleOrder（销售单，按门店独立）
Store ─── 1:N ──── PurchaseEntry（入库单，按门店独立）
Store ─── 1:N ──── Transfer（调货单，来源+目标门店）

Brand（品牌） ─── 1:N ──── Model（型号） ─── 1:N ──── SKU（单品，全门店共享）
SKU ──── Inventory（库存） ──── Store
Transfer ─── Store（来源） + Store（目标）
```

## 6. 关键模块详细设计

### 6.1 门店数据隔离机制

系统核心机制：**所有业务数据通过 `store_id` 隔离**。

```
用户登录
  → 获取当前用户 store_id
  → 所有 API 查询自动加上 WHERE store_id = ?
  → 新增数据自动填充 store_id
  → 超级管理员（store_id = null）可传 store_id 参数查看任意门店

前端：
  → Pinia 中存储当前用户信息和当前门店
  → 超级管理员在界面右上角切换门店
  → 普通用户固定为本门店
```

### 6.2 调货流程设计

```
分店 A（库存不足）                分店 B（有库存）
    │                                  │
    ├── 发起调货申请                    │
    │   选择来源门店=分店B               │
    │   填写商品+数量                   │
    │   状态: pending                   │
    │                                  │
    │                                  ├── 看到待调出列表
    │                                  │   确认调出
    │                                  │   → 扣减 B 店库存
    │                                  │   → 状态: outbound
    │                                  │
    ├── 看到已调出列表                   │
    │   确认收货                        │
    │   → 增加 A 店库存                  │
    │   → 状态: done                    │
    │                                  │
    └── 调货完成 ✓                     │
```

### 6.3 调货后端事务

```
PUT /api/v1/transfers/1/outbound（来源门店确认调出）
BEGIN TRANSACTION
  ① 查询 wh_inventory WHERE sku_id=? AND store_id=来源店
  ② 校验库存 >= 调货数量
  ③ 扣减来源店库存: UPDATE quantity = quantity - ?
  ④ 写入库存流水 (change_type='transfer_out')
  ⑤ 更新调货单状态为 outbound
COMMIT

PUT /api/v1/transfers/1/inbound（目标门店确认收货）
BEGIN TRANSACTION
  ① 查询 wh_inventory WHERE sku_id=? AND store_id=目标店
  ② IF 存在 THEN UPDATE quantity = quantity + ?
     ELSE INSERT quantity = ?
  ③ 写入库存流水 (change_type='transfer_in')
  ④ 更新调货单状态为 done
COMMIT
```

### 6.4 销售开单 — 带门店标识

```
POST /api/v1/sales
{
  "sku_id": 1,
  "quantity": 1,
  "unit_price": 6999,
  "actual_amount": 6799,
  "store_id": 1,         // 自动从当前用户获取，非超级管理员不可修改
  "customer_name": "张先生"
}

后端事务：
BEGIN TRANSACTION
  ① 扣减本店库存: wh_inventory WHERE sku_id=? AND store_id=?
  ② 写入库存流水 (change_type='out', store_id=?)
  ③ 创建销售单 (store_id=?, order_no含门店编码)
COMMIT
```

### 6.6 期初库存录入设计

#### 6.6.1 业务场景

> 门店刚启用系统时，货架上已有大量手机库存。店员/管理员打开"期初库存录入"页面，选择SKU，填写现有数量，系统写入该门店的初始库存，作为系统启用的起点。

#### 6.6.2 录入流程

```
① 进入"期初库存录入"页面
         │
② 选择门店（默认当前用户所属门店，仅管理员可切换）
         │
③ 录入商品及数量
   ┌──────────────────────────────────────┐
   │ 方式A：逐条录入                       │
   │ 选择SKU → 填写数量 → 点击"添加"      │
   │ → 加入待确认列表                      │
   ├──────────────────────────────────────┤
   │ 方式B：批量录入                       │
   │ 一次选择多个SKU，分别填写数量          │
   │ → 一次性加入待确认列表                │
   └──────────────────────────────────────┘
         │
④ 确认待确认列表
   显示：SKU名称 | 数量 | 备注（可填来源说明）
         │
⑤ 点击"确认录入"
   ├── 后端事务操作：
   ├── 逐条写入 wh_inventory（如已有则覆盖）
   ├── 写入 wh_inventory_log（change_type = 'initial'）
   └── 返回录入结果（成功/失败明细）
```

#### 6.6.3 API 接口

```json
// POST /api/v1/inventory/initial
// 单条期初录入
{
  "store_id": 1,
  "sku_id": 1,
  "quantity": 10,
  "remark": "系统启用时盘点录入"
}

// 响应
{
  "code": 200,
  "data": {
    "sku_name": "iPhone 17 Pro Max 256GB 深黑",
    "quantity": 10,
    "change_type": "initial"
  }
}
```

```json
// POST /api/v1/inventory/initial/batch
// 批量期初录入
{
  "store_id": 1,
  "items": [
    { "sku_id": 1, "quantity": 10 },
    { "sku_id": 2, "quantity": 5 },
    { "sku_id": 3, "quantity": 8 }
  ],
  "remark": "2025年6月系统启用期初库存"
}

// 响应
{
  "code": 200,
  "data": {
    "total": 3,
    "success": 3,
    "failed": 0,
    "items": [

      { "sku_id": 2, "sku_name": "iPhone 17 Pro Max 512GB 白色", "quantity": 5, "status": "success" },
      { "sku_id": 3, "sku_name": "iPhone 17 128GB 蓝色", "quantity": 8, "status": "success" }

  }
}
```

#### 6.6.4 后端事务逻辑

```
POST /api/v1/inventory/initial/batch

BEGIN TRANSACTION
  FOR EACH item IN items:
    ① 查询 wh_inventory WHERE sku_id=? AND store_id=?
    ② IF 存在 THEN
         UPDATE quantity = quantity + item.quantity
         old_qty = inv.quantity
       ELSE
         INSERT (sku_id, store_id, quantity) VALUES (?, ?, item.quantity)
         old_qty = 0
       END IF
    ③ 写入 wh_inventory_log (
         change_type = 'initial',
         qty_before = old_qty,
         qty_change = item.quantity,
         qty_after = old_qty + item.quantity
       )
  END FOR
COMMIT
```

> 设计说明：如果该SKU在门店已有库存（如之前已录入或入库过），则累加数量而非覆盖。保证不会因为误操作清零已有库存。

## 9. 数据流设计

### 9.1 店内操作数据流

```
[本店进货] 采购入库 → 本店库存增加
[本店销售] 扫码开单 → 扣本店库存 → 打印小票
[库存盘点] 创建盘点 → 审核 → 调整本店库存
[门店调货] 发起申请 → 来源店出库 → 目标店入库
```

### 9.2 门店间操作数据流

```
分店A                             分店B
  │                                  │
  │ ① 销售开单                        │ ① 销售开单
  │ 扣A店库存 ✓                       │ 扣B店库存 ✓
  │                                  │
  │ ② 发起调货（从B店调iPhone 17）    │
  │ ──────────────────────────────────→ │
  │                                  │ ③ 确认调出
  │                                  │ 扣B店库存
  │ ←────────────────────────────────── │
  │ ④ 确认收货                        │
  │ A店库存增加 ✓                      │
```

### 9.3 超级管理员数据流

```
超级管理员登录
  ├── 默认查看全部门店汇总数据
  ├── 可切换门店查看单个门店数据
  ├── 销售统计：按门店筛选/对比
  ├── 库存统计：查看各门店库存
  └── 调货记录：查看所有调货单
```

## 11. 部署方案（Tailscale 组网）

### 11.1 前置准备 — 安装 Tailscale

> Tailscale 是一款基于 WireGuard 的虚拟组网工具，在 [tailscale.com](https://tailscale.com) 注册账号后，在每台电脑上安装客户端即可自动组网。

```
每台电脑都需要：
① 注册 Tailscale 账号（Google/Microsoft/Apple 账号均可）
② 在各电脑上安装 Tailscale 客户端
   Windows: 下载安装 Tailscale for Windows
③ 登录同一账号，各电脑自动分配 100.x.x.x 虚拟IP
④ 在任意一台电脑上 ping 另一台的 Tailscale IP，确认互通
```

> Tailscale 免费版支持 3 个用户/设备，刚好覆盖分店A + 分店B + 老板的场景。

### 11.2 服务器电脑部署

```
1. 选择任意一台门店电脑作为服务器（建议配置较好的）
2. 确认该电脑已安装 Tailscale 并在线
3. 安装 Node.js 运行环境（一次性）
4. 复制项目文件夹，双击 start.bat
5. 后端监听 0.0.0.0:3000（Tailscale 虚拟网可访问）
6. 首次启动自动创建 SQLite 数据库
```

### 11.3 各门店电脑访问

```
1. 确认各电脑已安装 Tailscale 并在线
2. 确认可以 ping 通服务器电脑的 Tailscale IP
3. 打开浏览器访问 http://服务器TailscaleIP:3000
   例如：http://100.1.2.3:3000
4. 用本店账号登录，自动进入本店界面
5. CLodop 安装在需要打印的电脑上
```

### 11.4 启动脚本

```bash
# start.bat (Windows) — 在服务器电脑上运行
@echo off
cd /d %~dp0
npm install
set HOST=0.0.0.0
set PORT=3000
npm run start
```
