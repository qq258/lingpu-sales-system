# 无库存销售功能 - 设计文档

## 概述

在 portal 的销售开单页面（Sale.vue）增加"无库存销售"模式，用于处理历史商品在售出时尚未登记入库的场景。该模式下允许手动录入品牌型号、扫码录入 IMEI，收款后自动补建品牌型号和库存记录。

## 1. 销售类型切换

在页面顶部、"扫描 IMEI 码"区域上方，新增 radio 切换组件：

- 选项：`普通销售` / `无库存销售`
- 默认选中"普通销售"
- 切换时若购物车有商品，清空购物车并提示用户
- 状态管理：`const saleType = ref<'normal' | 'no-stock'>('normal')`
- 切换后条件渲染不同的输入区域

## 2. 品牌型号搜索区域（无库存销售模式）

### 2.1 品牌型号输入框
- 一个输入框，placeholder="搜索或输入品牌型号"
- 监听 `input` 事件，300ms 防抖
- 调用 `GET /products/models?keyword=xxx` 搜索型号（后端新增 `keyword` 模糊匹配参数，匹配品牌名和型号名）
- 如果有已选门店，可过滤当前门店库存中已有的型号

### 2.2 匹配结果下拉框
- 输入框下方弹出匹配结果列表
- 显示格式：`品牌 型号 - 颜色 / 存储 (¥售价)`
- 支持键盘 ↑↓ 导航 + Enter 选择，或点击选择
- 选择后输入框显示完整文本并锁定，显示已选标签

### 2.3 无匹配时
- 下拉列表底部显示"未找到匹配项，将保存为新品牌型号"
- 用户可直接确认输入的文本，进入 IMEI 录入阶段

### 2.3.1 提前保存品牌型号
- 在提示文字右侧增加"保存"按钮
- 点击后打开 SaveBrandDialog 弹窗（同 4.2），用户确认后调用接口新增品牌型号
- 保存成功后该商品的品牌型号状态标记为"已保存"
- 已保存的品牌型号在步骤 4 中遍历时自动跳过，无需重复新增

### 2.4 IMEI 录入区域
- 品牌型号确认后自动展开
- 上方增加 **售出价格** 输入框，自动填充匹配到的型号售价（如有），可手动修改
- 下方三个输入框：IMEI1、IMEI2、SN
- 支持扫码枪逐字段录入，IMEI1 录入完自动聚焦到 IMEI2，再到 SN
- 录入完成后点击"添加至购物车"

## 3. 购物车与结算

### 3.1 购物车列表（CartItemCard 改造）
- 每个商品展示：品牌型号、颜色/存储、IMEI1 / IMEI2 / SN、售价
- 三个 IMEI/SN 字段初始值为录入区带过来的值
- 未录入的字段显示可点击的 `[待录入]` 标签，点击后行内变为输入框
- 输入框支持扫码枪录入，自动跳转焦点
- 已录入的字段显示值

### 3.2 售价
- 匹配到已有型号 → 自动填入该型号售价到 2.4 的售出价格输入框，允许修改
- 新品牌型号 → 在 2.4 的售出价格输入框中手动输入
- 添加到购物车时售出价格自动带入购物车商品行

### 3.3 结算区域
- 与普通销售一致：合计金额、实收金额、找零、客户姓名、客户电话、客户地址、备注

## 4. 确认收款后流程

### 4.1 品牌型号分割逻辑
- 遍历购物车中每个无库存商品的品牌型号文本
- 按第一个空格分割：`"苹果 iPhone 15"` → 品牌 `"苹果"`、型号 `"iPhone 15"`
- 检查分割出的品牌是否在 `pdt_brand` 中存在

### 4.2 新增品牌型号弹窗
- 组件：`SaveBrandDialog.vue`
- **手风琴展示**：每个需要新增的商品一个面板
- 面板标题：用户手动录入的原始文本
- 面板内容：品牌和型号两个输入框（可编辑），预填分割结果
- 手风琴默认展开第一个
- **仅"确认"按钮**，无取消按钮（必须完成保存）
- 确认后逐个调接口新增

### 4.3 后端新增接口 `POST /sales/no-stock`

在 `server/src/routes/sales.ts` 中新增路由：

**请求体：**
```json
{
  "store_id": 1,
  "items": [
    {
      "brand_name": "苹果",
      "model_name": "iPhone 15 Pro Max",
      "model_id": null,
      "color": "",
      "storage": "",
      "imei": "123456789012345",
      "imei2": null,
      "sn_code": null,
      "unit_price": 8999
    }
  ],
  "actual_amount": 8999,
  "customer_name": "",
  "customer_phone": "",
  "customer_address": "",
  "remark": ""
}
```

**事务内逻辑：**
1. 如果 `model_id` 为 null，根据 `brand_name` 查找或创建品牌
2. 根据 `brand_id` + `model_name` 查找或创建型号（sale_price 取 `unit_price`）
3. 在 `wh_inventory_imei` 创建 IMEI 记录，状态为 `sold`
4. 更新 `wh_inventory` 汇总库存（增加后立即减少，或 net 0 但确保记录存在）
5. 创建 `sale_order` + `sale_order_item`

### 4.4 前端组合调用流程

```
确认收款 → 遍历购物车，检查哪些商品的品牌型号需要新增
       ↓
   有需要新增的 → 打开 SaveBrandDialog 弹窗 → 用户确认
       ↓                                         ↓
   都无需新增                               调接口逐个创建品牌/型号
       ↓                                         ↓
   调 POST /sales/no-stock 完成销售 ←────────────┘
       ↓
   打印小票 → 清空购物车
```

## 5. 组件结构

```
Sale.vue
├── 销售类型 Radio 切换
├── [普通销售] 扫描 IMEI 区域（现有）
├── [无库存销售] 品牌型号搜索区域（新增）
│   ├── 品牌型号输入 + 下拉匹配列表
│   └── IMEI1/IMEI2/SN 录入区
├── 购物车列表
│   └── CartItemCard（改造，支持 IMEI 行内编辑）
├── 结算区域（客户信息 + 收款）
├── ConfirmDialog（确认收款）
├── SaveBrandDialog（新增品牌型号弹窗，新增）
└── PrintReceipt（打印小票）
```

## 6. 后端变更

| 路由 | 方法 | 变更 | 说明 |
|------|------|------|------|
| `/products/models` | GET | 扩展 | 新增 `keyword` 查询参数，模糊匹配品牌名和型号名 |
| `/sales/no-stock` | POST | 新增 | 无库存销售（含品牌型号创建 + IMEI 录入 + 销售订单） |

## 7. 普通销售 → 无库存销售跳转

当用户处于普通销售模式、扫码查询 IMEI 未匹配到库存时：
- 当前行为：显示错误提示"未找到该IMEI对应的库存记录"
- **新增行为**：显示错误提示的同时，弹窗询问用户："未找到该 IMEI 的库存记录，是否切换到无库存销售模式？"
  - 用户选择"是" → 自动切换到无库存销售模式，并将当前扫描的 IMEI 填入 IMEI1 输入框，聚焦到品牌型号输入框
  - 用户选择"否" → 关闭弹窗，清空扫描输入框，保持普通销售模式
- 调用 `scanImeiForSale` 接口返回 404 时触发此判断
- 切换时会清空购物车（与手动切换行为一致）

### 7.1 交互流程

```
普通销售扫码 → IMEI 未匹配 → 弹窗"未找到库存，是否切换到无库存销售？"
       ↓                    ↓
   选"否" → 清空输入，继续普通销售
       ↓
   选"是" → 切换 saleType 为 'no-stock'
          → 当前 IMEI 自动填入 IMEI1 录入框
          → 聚焦到品牌型号输入框
          → 用户补全品牌型号信息

## 8. 数据流

```
用户输入品牌型号 → 前端搜索匹配 → 录入 IMEI → 加入购物车
       ↓
确认收款 → 品牌型号检查 → 弹窗确认 → 后端 no-stock 接口
       ↓
  打印小票 → 完成
```
