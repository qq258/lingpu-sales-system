# 手机销售后台管理系统 — 需求规格说明书

## 1. 引言

### 1.1 编写目的
本文档旨在明确手机销售后台管理系统的功能需求、非功能需求及约束条件，为后续系统设计、开发、测试和验收提供依据。本文档面向项目团队（产品、开发、测试）及相关干系人。

### 1.2 项目背景
某连锁手机销售门店拥有两个分店（异地），日常需要管理手机产品的进货、库存、销售及门店间调货。两个分店通过 **Tailscale** 组建虚拟局域网，需要建设一套统一的后台管理系统，部署在其中一台电脑上，两个分店通过 Tailscale 虚拟IP访问使用。系统需支持多门店数据隔离（各门店只能操作自己的数据），同时老板可查看全部门店的经营数据，并支持门店间调货。

### 1.3 适用范围
本系统为 B/S 架构的后台管理系统，部署在 Tailscale 虚拟局域网内，适用于连锁手机销售门店的多分店统一管理。

### 1.4 术语与定义

| 术语 | 说明 |
|------|------|
| SKU | 库存量单位，用于唯一标识一款手机的具体配置（颜色、存储等） |
| 厂家条码 | 手机出厂时印在包装盒上的 EAN-13/UPC 条形码 |
| IMEI | 国际移动设备识别码，手机机身唯一条码，每台手机唯一 |
| CLodop | 云打印中间件，用于实现 Web 端直接打印小票 |
| 扫码枪 | USB/HID 条码扫描设备，扫描手机包装盒条码自动识别型号 |
| Tailscale | 基于 WireGuard 的虚拟组网工具，将异地的电脑组成一个虚拟局域网 |
| 调货 | 将一个门店的库存转移到另一门店 |

---

## 2. 总体需求

### 2.1 业务目标
- 实现一套系统统一管理所有分店业务
- 各门店数据隔离，店员仅操作本店数据
- 老板/管理员可查看全部门店数据
- 支持门店间调货，调货后双方库存自动更新
- 扫码枪贯穿三大环节：创建SKU、采购入库、销售开单
- 支持 CLodop 小票打印
- 提供跨门店的数据看板与销售统计

### 2.2 用户角色

| 角色 | 描述 | 主要权限范围 |
|------|------|-------------|
| **超级管理员** | 老板/系统所有者 | 全部功能；用户管理、门店管理、查看全部门店数据、调货审核 |
| **店长** | 各分店负责人 | 本店全部功能；本店用户管理、调货申请/接收 |
| **店员** | 日常业务操作员 | 本店产品管理、进货、销售、库存查询、打印小票、统计查看 |

> 每个用户绑定所属门店（超级管理员不绑定，可查看所有门店）。

## 3. 功能需求

### 3.1 基础鉴权系统

#### 3.1.1 用户登录
- 支持用户名/密码登录
- 密码需加密存储
- 登录后自动识别所属门店，界面仅展示该门店数据（超级管理员可选门店）

#### 3.1.2 用户管理
- 用户的新增、编辑、删除、查询
- 支持启用/禁用账户
- 重置用户密码
- 用户绑定所属门店（超级管理员/店长/店员）

### 3.2 门店管理

#### 3.2.1 门店设置
- 门店的新增、编辑、删除
- 字段：门店名称、编码、地址、联系电话、状态

---

### 3.3 产品数据管理（核心 P0）

#### 3.3.1 品牌管理
- 品牌的新增、编辑、删除、查询
- 品牌字段：名称、描述

#### 3.3.2 手机型号管理
- 型号的新增、编辑、删除、查询
- 型号关联品牌
- 字段：型号名称、品牌、上市年份、操作系统、网络制式、屏幕尺寸、CPU、电池容量、描述等

#### 3.3.3 SKU 管理（核心）
- SKU 的新增、编辑、删除、查询
- SKU 关联手机型号
- 字段：SKU 编码（自动生成）、**厂家条码（EAN-13）**、颜色、配置（RAM/ROM）、销售价格、进货成本价、状态
- 支持上传产品展示图片（本地文件系统）
- **SKU 为基础数据，全门店共享，无需各自录入**

##### 厂家条码说明
- 每台手机出厂时，包装盒上印有 EAN-13 条形码
- 创建 SKU 时扫码录入厂家条码
- 三个环节共用同一条码：录入SKU时登记 → 入库时扫码匹配 → 销售时扫码识别
- 条码字段需建立唯一索引

---

### 3.4 进货管理（核心 P0）

#### 3.4.1 供应商管理
- 供应商的新增、编辑、删除、查询
- 字段：供应商名称、联系人、联系电话、地址、备注

#### 3.4.2 采购入库 — 逐台录入 IMEI

每台手机有唯一的 IMEI 码（国际移动设备识别码），入库时**必须以"台"为单位逐条录入**，不能只增加数量。

##### 业务场景描述
> 从供应商处拿到一批手机 → 打开系统新建入库单，选择供应商 → 用扫码枪逐台扫描手机包装盒上的厂家条码（识别型号）→ 继续逐台扫描手机机身上的 IMEI 条码 → 系统逐条记录每台手机的 IMEI 信息 → 全部录入完毕后确认入库 → 系统将每台 IMEI 标记为"在库"，并增加对应 SKU 的库存数量

##### 入库操作流程

```
① 新建入库单 → 选择供应商、填写入库日期
         │
② 开始逐台录入手机（每台一条记录）
   ┌──────────────────────────────────────────┐
   │ 每台手机操作步骤：                        │
   │ 步骤一：扫描/选择 SKU（扫包装盒条码）     │
   │          → 自动识别型号、颜色、配置       │
   │ 步骤二：录入 IMEI（扫机身条码/手动输入）  │
   │          → IMEI 唯一性校验               │
   │ 步骤三：点击"添加" → 加入本次入库列表     │
   │                                          │
   │ 重复以上步骤，直到所有手机录入完毕          │
   └──────────────────────────────────────────┘
         │
③ 查看本次入库汇总列表
   ┌──────────────────────────────────────────┐
   │ 序号 │ SKU           │ IMEI              │
   │ 1   │ iPhone 17 256G │ 356938123456789   │
   │ 2   │ iPhone 17 256G │ 356938123456790   │
   │ 3   │ iPhone 17 512G │ 356938123456791   │
   │ ... │               │                    │
   │ 总计：10 台                              │
   └──────────────────────────────────────────┘
         │
④ 确认入库
   ├── 逐条校验 IMEI 唯一性（系统全局不可重复）
   ├── 逐条写入 wh_inventory_imei（标记 status='in_stock'）
   ├── 按 SKU 汇总增加 wh_inventory 数量
   ├── 写入库存流水
   └── 生成入库单号
```

##### 关键设计
- **IMEI 为必填项**：每台手机必须有 IMEI，不可跳过
- **IMEI 全局唯一**：同一 IMEI 不可两次入库（拦截重复录入）
- **IMEI 录入方式**：扫码枪扫描机身条码（优先） / 手动输入
- **一单一品**：一个入库单项 = 1 台手机（一个 SKU + 一个 IMEI）
- **入库确认后**：IMEI 写入 `wh_inventory_imei` 表，status = `in_stock`

---

### 3.5 库存管理（核心 P0）

#### 3.5.1 期初库存录入（手动录入）
> 系统上线前门店已有库存，需要在系统中录入初始库存数据，作为系统启用的基础。

- **功能入口**：库存管理 → 期初库存录入
- **操作方式**：选择门店（默认本店）、选择SKU、填写初始数量、填写备注
- **支持逐条录入和批量录入**（一次填多个SKU）
- **录入确认后**，系统自动写入 `wh_inventory` 库存表，并标记数据来源为 `initial`
- **写入库存流水**：记录 `change_type = 'initial'`，表示为期初入库
- **期初库存记录仅限管理员操作**，录入后不可修改（如需调整需通过盘点流程）

#### 3.5.2 库存查询
- 各门店查看本店实时库存
- 超级管理员可查看全部库存（按门店筛选）
- 支持按 SKU、IMEI 查询
- 查看库存变动明细（含期初、入库、销售出库、调货、盘点等类型）

#### 3.5.3 库存盘点
- 创建盘点单，录入盘点数量，自动计算盈亏差异
- 盘点审核后调整本店实际库存

---

### 3.6 门店间调货（核心新增 P0）

#### 3.6.1 业务场景
> 分店 A 的客户预定了 iPhone 17，但 A 店库存不够，B 店还有库存 → A 店发起调货申请 → 选择从 B 店调货 → **在 B 店选择具体要调出的 IMEI** → B 店确认调出（标记 IMEI 为 transfer_out）→ A 店确认收货（标记 IMEI 为 in_stock）

#### 3.6.2 调货申请
- 发起调货：选择**来源门店**、选择商品（SKU）、**系统列出来源门店该 SKU 的所有可用 IMEI**、勾选要调货的 IMEI
- 系统自动校验来源门店 IMEI 是否可用
- 调货单状态流转：待调出 → 已调出 → 已完成 → 已取消

#### 3.6.3 调货出库（来源门店操作）
- 来源门店看到"待调出"的调货单
- 确认调出后，来源门店对应 IMEI 标记为 `transfer_out`（调出中），库存数量扣减

#### 3.6.4 调货入库（目标门店操作）
- 目标门店看到"已调出"的调货单
- 确认收货后，**目标门店新增这些 IMEI 记录**，状态为 `in_stock`，库存数量增加
- 写入库存流水记录（标记为调货类型）

#### 3.6.5 调货记录查询
- 支持按时间、门店、商品维度查询调货历史

---

### 3.7 销售开单（核心 P0）

#### 3.7.1 业务场景
> 客户进店挑选手机 → 扫码开单 → 调价 → 收款 → 扣本店库存 → 打印小票

#### 3.7.2 销售开单
- **商品选择**：扫码或手动搜索 SKU
- **IMEI 选择**：自动列出该 SKU 在本店的所有可用 IMEI（status='in_stock'），勾选要出售的 IMEI，数量自动匹配勾选的 IMEI 个数
- **销售价**：自动带入预设价格，可手动修改
- **数量**：等于已选 IMEI 个数，不可手动修改
- **应收/实收**：自动计算，实收可修改
- **确认保存**：
  - 自动生成订单编号（含门店标识）
  - 自动填充销售时间
  - **扣减本店库存**（按数量扣减）
  - **逐条标记 IMEI 状态为 'sold'**
  - 保存后弹窗询问是否打印小票

#### 3.7.3 打印小票
- 保存后弹窗"是否打印小票？"
- 支持补打
- 小票内容含门店名称

#### 3.7.4 销售记录
- 按时间倒序展示本店销售记录
- 超级管理员可按门店筛选查看所有销售记录

---

### 3.8 数据统计与看板

#### 3.8.1 数据看板（首页）
- **店员/店长**：展示本店今日销售额、今日单数、近7日趋势、最近订单
- **超级管理员**：展示全部门店汇总数据，支持按门店筛选查看

#### 3.8.2 销售统计
- 按日/周/月维度统计销售额、销售单数
- 热销型号排行
- 支持按门店筛选、门店间对比

#### 3.8.3 库存统计
- 各门店库存总值
- 低库存预警

#### 3.8.4 调货统计
- 调货记录查询与导出
- 各门店调出/调入汇总

---

## 4. 非功能需求

### 4.1 性能需求
- 页面响应时间：常规操作 ≤ 2 秒
- 局域网环境，2-3 个并发用户

### 4.2 安全性
- 密码 bcrypt 加密存储
- 数据隔离：各门店只能查看和操作本店数据
- 敏感操作记录操作日志

### 4.3 可用性
- 界面简洁清晰
- 销售开单 3 步内完成
- 关键操作二次确认
- **界面右上角显示当前门店，超级管理员可一键切换门店视角**

### 4.4 可维护性
- SQLite 数据库，单文件易备份
- 一键备份与恢复

### 4.5 兼容性
- 支持 Chrome、Edge 最新版本
- 适配 1920×1080 及以上分辨率

---

## 5. 约束条件

- **部署方式**：服务器电脑部署后端+数据库，所有电脑安装 Tailscale 组网，通过 Tailscale 虚拟 IP 访问
- **数据库**：SQLite，存储在服务器电脑上
- **打印**：CLodop 插件安装在**每台需要打印的电脑**上（含服务器电脑和各门店电脑）
- **网络**：各分店通过 Tailscale 组成虚拟局域网，无需公网 IP，无需端口映射
- **技术栈**：前后端一体化轻量方案
